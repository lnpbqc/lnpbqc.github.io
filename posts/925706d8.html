核心架构：为什么一个注解能管理全局事务

一、@GlobalTransactional 到底做了什么？

这个注解本身并不直接管理其他注解，而是开启了一个上下文（Context），后续所有参与者的行为都被这个上下文所影响：

总结

你不需要手动配置任何东西，Seata starter 依赖引入后：

  1. Feign：自动包装 SeataFeignClient，发送请求前自动注入 TX_XID Header
  2. Dubbo：通过 SPI 自动加载 TransactionPropagationFilter，消费端注入/服务端提取 Attachment

这就是”约定优于配置”——框架自动识别你的通信方式并代理改写。

四、总结：一个注解的背后

层面 机制 是否需要你写注解
事务发起 @GlobalTransactional ✅ 需要（TM）
事务参与 DataSourceProxy 自动拦截 ❌ 不需要（RM）
跨服务传播 Feign/Dubbo 拦截器自动注入 XID ❌ 不需要
分支注册 SQL 执行时自动向 TC 注册 ❌ 不需要
UndoLog 自动解析 SQL 生成前后镜像 ❌ 不需要
二阶段提交 TC 自动驱动所有 RM ❌ 不需要

核心设计理念：Seata 采用 “约定优于配置” + AOP 代理 的方式，将分布式事务的复杂性封装到底层。你只需要在事务发起方标记 @GlobalTransactional，其余参与者通过资源代理和上下文传播自动加入全局事务。

如果你需要，我可以进一步展开：

  1. UndoLog 的具体生成机制（SQL 解析、镜像生成）
  2. TC 的二阶段协调源码分析
  3. 与 Spring @Transactional 的共存策略

一句话总结

XID 是全局事务的”身份证”，通过 ThreadLocal 在同服务内传递，通过 HTTP/RPC Header 跨服务传递，所有参与者凭这个 ID 与 TC 协调。

那他们各个服务需要用本地事务进行管理吗？

不需要显式加本地事务注解，但 Seata 底层实际上帮你做了类似本地事务的事情。

Seata AT 模式下，分支事务的本地事务由 DataSourceProxy 自动管理，不需要也不应该加 @Transactional。

TC 怎么知道全局事务什么时候结束？

TM（@GlobalTransactional 拦截器）包裹了整个业务方法，方法执行完毕（正常返回或异常抛出）时，TM 主动通知 TC”结束了”。TC 自己不会感知，全靠 TM 汇报。
