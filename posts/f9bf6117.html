第二章：表结构设计

2.1 表的概念

表是数据库中存储数据的基本单位。一个表由行和列组成：行表示一条记录，列表示数据的属性。例如，商品表中每一行代表一件商品，每一列代表商品的属性（如商品名称、价格等）。

TEXT ：长文本，最大65535字节，适合存储文章、描述等长文本内容。

对比 ：

  • CHAR 适合存储固定长度的数据（如性别”男”/“女”、国家代码），查询速度快但浪费空间。
  • VARCHAR 适合存储长度不固定的数据（如姓名、地址），节省空间但查询稍慢。

第三章：数据约束

3.1 约束的作用

数据约束用于保证数据的完整性和一致性。通过约束，可以防止无效数据进入数据库，确保数据符合业务规则。常见约束包括主键、非空、唯一、默认值、检查约束和外键约束。

3.2 主键约束（PRIMARY KEY）

主键用于唯一标识表中的每一行数据。主键列的值不能重复，不能为空。每个表只能有一个主键，但主键可以由单列或多列组成。

复合主键要求多列的组合值唯一，单列值可以重复，但组合值必须唯一。例如 (order_id=1, goods_id=1) 和 (order_id=1, goods_id=2) 可以同时存在，但不能有两个 (order_id=1, goods_id=1)。

3.3 自增长（AUTO_INCREMENT）

AUTO_INCREMENT 用于自动生成唯一递增的整数值，通常与主键配合使用。自增长列默认从1开始，每次插入新记录时自动加1。

3.4 非空约束（NOT NULL）

NOT NULL 约束要求列的值不能为空。适合必填字段，如用户名、商品名称等。

未指定 NOT NULL 的列默认允许 NULL 值。

3.5 默认值（DEFAULT）

DEFAULT 约束为列指定默认值。当插入数据时未提供该列的值，系统自动使用默认值。

3.6 唯一约束（UNIQUE）

UNIQUE 约束要求列的值不能重复，但允许多个 NULL 值。适合需要唯一但可选的字段，如邮箱、身份证号等。

email 设置为 UNIQUE 且 NOT NULL，不允许重复也不允许为空。card_id 仅设置 UNIQUE，可以有多个 NULL 值。

3.7 检查约束（CHECK）

CHECK 约束用于限制列的取值范围。只有满足检查条件的数据才能插入。

注意 ：MySQL 5.7 不支持 CHECK 约束，定义时不会报错，但约束不会生效。MySQL 8.0 开始支持 CHECK 约束。

3.8 外键约束（FOREIGN KEY）

外键用于建立表之间的关联关系，保证数据的参照完整性。外键列的值必须在主表的主键或唯一列中存在，或为 NULL。

3.8.1 外键语法

外键要求：

  • 外键必须引用主表的 PRIMARY KEY 或 UNIQUE 列
  • 主表和从表必须使用 InnoDB 引擎
  • 外键列的数据类型必须与主表被引用列的类型一致

3.8.2 外键作用

外键建立后，主表中被引用的记录无法删除，除非没有从表记录引用它。这保证了数据的一致性，防止出现”孤儿记录”。

如果需要在删除主表记录时自动处理从表数据，可以使用级联操作：

  • ON DELETE CASCADE：删除主表记录时自动删除从表中的相关记录
  • ON DELETE SET NULL：删除主表记录时将从表外键设置为NULL
  • ON UPDATE CASCADE：更新主表主键时自动更新从表外键

第八章：索引优化

8.1 索引概念

索引是一种数据结构，用于加快数据库查询速度。类似于书籍的目录，索引允许数据库快速定位数据，而不需要扫描整个表。

索引的本质是用空间换时间。创建索引会占用额外的存储空间，但可以显著提高查询性能。对于频繁查询的大表，索引是必不可少的优化手段。

8.2 索引原理

当执行查询时，如果没有索引，数据库需要进行全表扫描，逐行检查是否满足条件。对于百万级数据，全表扫描非常耗时。

索引通过建立数据结构（如B树、B+树）来组织数据。当查询使用索引列时，数据库可以直接通过索引定位到目标数据，大大减少了需要扫描的行数。

8.3 索引类型

MySQL支持多种索引类型，适用于不同的场景。

8.3.1 主键索引（PRIMARY KEY）

主键索引是最常用的索引类型。当定义主键时，数据库自动创建主键索引。

特点 ：

  • 自动创建，无需手动添加
  • 值必须唯一且非空
  • 每个表只能有一个主键索引

8.3.2 唯一索引（UNIQUE）

唯一索引用于保证列值的唯一性。当定义 UNIQUE 约束时，数据库自动创建唯一索引。

特点 ：

  • 列值必须唯一，但允许 NULL
  • 可以有多个唯一索引
  • 自动创建，也可手动创建

email 列会自动创建唯一索引，查询时可以利用索引加速。

8.3.3 普通索引（INDEX）

普通索引是最常用的手动索引类型，用于加速查询。

特点 ：

  • 列值可以重复
  • 可以为多个列创建索引
  • 需要手动创建和删除

普通索引适用于频繁出现在 WHERE、ORDER BY、JOIN 条件中的列。

8.3.4 全文索引（FULLTEXT）

全文索引用于全文搜索，适合在文本列中搜索关键词。

特点 ：

  • 仅适用于 MyISAM 存储引擎（MySQL 5.6后InnoDB也支持）
  • 适用于 CHAR、VARCHAR、TEXT 类型
  • 用于全文检索，支持自然语言搜索

注意 ：对于生产环境的全文搜索，推荐使用专业的全文搜索引擎，如 Solr 或 ElasticSearch，它们提供更强大的搜索功能和更好的性能。

8.7 何时使用索引

索引不是越多越好，需要根据实际查询场景合理创建。

8.7.1 适合创建索引的情况

1. 频繁出现在 WHERE 条件中的列

2. 频繁用于 ORDER BY 排序的列

3. 频繁用于 JOIN 连接的列

4. 频繁用于 GROUP BY 分组的列

8.7.2 不适合创建索引的情况

1. 小表

如果表的数据量很小（如几百行），全表扫描的速度已经足够快，创建索引反而增加维护成本。

2. 频繁更新的列

索引需要在数据更新时同步维护。如果列频繁被 UPDATE，索引的维护成本会很高。

3. 重复值较多的列

如果列的值重复率很高（如性别只有”男”/“女”），索引的筛选效果不明显，不建议创建索引。

8.8 索引的代价

1. 占用存储空间

第九章：事务管理

9.1 事务概念

事务是一组相关的SQL操作，这些操作作为一个整体执行：要么全部成功，要么全部失败。事务用于保证数据的一致性和完整性。

典型场景 ：银行转账

假设张三向李四转账1000元，需要执行两个操作：

  1. 从张三账户扣除1000元
  2. 向李四账户增加1000元

这两个操作必须作为一个整体执行。如果只完成第一步，张三的钱扣了但李四没收到，数据就不一致了。事务可以保证两个操作要么都成功，要么都失败回滚。

9.3 ACID特性

事务具有四个重要特性，简称 ACID。

9.3.1 原子性（Atomicity）

原子性保证事务中的所有操作要么全部执行成功，要么全部失败回滚，不存在部分成功的情况。

示例：订单创建

如果 INSERT 失败（如外键约束不满足），UPDATE 操作也会被回滚，库存不会减少。

9.3.2 一致性（Consistency）

一致性保证事务执行前后，数据库从一个一致性状态转换到另一个一致性状态。数据的完整性约束不会被破坏。

示例 ：转账前后总金额不变，订单创建后库存与订单数量匹配。

9.3.3 隔离性（Isolation）

隔离性保证并发执行的事务相互独立，一个事务的中间状态不会被其他事务看到。

多个用户同时操作数据库时，每个事务都感觉像是在独立运行，不会互相干扰。

9.3.4 持久性（Durability）

持久性保证事务一旦提交，对数据库的修改就是永久性的，即使系统故障也不会丢失。

9.4 并发问题

当多个事务并发执行时，可能出现以下问题。

9.4.1 脏读（Dirty Read）

脏读是指一个事务读取了另一个事务未提交的数据。如果另一个事务回滚，读取的数据就是无效的。

场景示例 ：

时间 事务A 事务B
T1 START TRANSACTION
T2 UPDATE goods SET stock = 100 WHERE goods_id = 1
T3 START TRANSACTION
T4 SELECT stock FROM goods WHERE goods_id = 1; (读到100)
T5 ROLLBACK
T6 COMMIT

事务B读到了事务A未提交的数据（100），但事务A最终回滚了，所以事务B读到的是”脏数据”。

9.4.2 不可重复读（Non-repeatable Read）

不可重复读是指一个事务内多次读取同一数据，结果不一致。这是因为其他事务在两次读取之间修改或删除了数据。

场景示例 ：

时间 事务A 事务B
T1 START TRANSACTION
T2 SELECT unitprice FROM goods WHERE goods_id = 1; (读到5999)
T3 START TRANSACTION
T4 UPDATE goods SET unitprice = 6999 WHERE goods_id = 1;
T5 COMMIT
T6 SELECT unitprice FROM goods WHERE goods_id = 1; (读到6999)
T7 COMMIT

事务A两次读取同一商品价格，结果不同。

9.4.3 幻读（Phantom Read）

幻读是指一个事务内多次查询，结果集的行数不一致。这是因为其他事务在两次查询之间插入了新数据。

场景示例 ：

时间 事务A 事务B
T1 START TRANSACTION
T2 SELECT COUNT(*) FROM goods WHERE category = 1; (结果10)
T3 START TRANSACTION
T4 INSERT INTO goods VALUES (101, ‘New Product’, 999, 1, ‘Supplier’);
T5 COMMIT
T6 SELECT COUNT(*) FROM goods WHERE category = 1; (结果11)
T7 COMMIT

事务A两次统计同一分类的商品数量，结果不同。

9.5 事务隔离级别

MySQL提供四种事务隔离级别，用于控制并发事务的可见性，解决上述并发问题。

9.5.1 隔离级别对比

隔离级别 脏读 不可重复读 幻读 加锁读
读未提交（READ UNCOMMITTED） √ √ √ ×
读已提交（READ COMMITTED） × √ √ ×
可重复读（REPEATABLE READ） × × × ×
可串行化（SERIALIZABLE） × × × √

说明 ：

  • √ 表示可能发生该问题
  • × 表示不会发生该问题

9.5.2 读未提交（READ UNCOMMITTED）

最低的隔离级别。事务可以读取其他事务未提交的数据。

特点 ：

  • 可能发生脏读、不可重复读、幻读
  • 性能最好，但数据一致性最差
  • 实际应用中很少使用

9.5.3 读已提交（READ COMMITTED）

事务只能读取其他事务已提交的数据。

特点 ：

  • 避免脏读
  • 可能发生不可重复读、幻读
  • Oracle数据库的默认隔离级别

9.5.4 可重复读（REPEATABLE READ）

事务内多次读取同一数据，结果一致。

特点 ：

  • 避免脏读、不可重复读
  • MySQL通过MVCC机制也避免了幻读
  • MySQL的默认隔离级别

9.5.5 可串行化（SERIALIZABLE）

最高的隔离级别。强制事务串行执行，完全隔离。

特点 ：

  • 避免所有并发问题
  • 性能最差，并发度最低
  • 通过锁实现，会导致大量阻塞
